# ADR-002: 统一数据库为 PostgreSQL 16

## Status

Accepted — 已实施，后续数据库选型锁定为 PostgreSQL 16。

## Context

项目早期阶段，不同环境使用了不同的数据库，导致开发和部署过程中出现不一致问题：

- **README 和部分文档** 提到 MySQL 作为数据库。
- **docker-compose.yml** 使用 `postgres:16-alpine` 镜像，连接串为 `postgresql://shopminer:shopminer@db:5432/shopminer`。
- **开发环境** 部分成员使用 SQLite 以便快速本地启动（无需安装数据库服务）。

这种混用局面带来的具体问题包括：

1. **SQL 方言差异**: SQLite 不支持 `FOR UPDATE` 行级锁（订单并发控制依赖此特性），且日期函数（`strftime` vs `DATE_TRUNC`）行为不同，导致代码在 SQLite 开发环境通过、PostgreSQL 生产环境报错。
2. **迁移脚本不一致**: Alembic 迁移在 SQLite 和 PostgreSQL 上生成不同的列类型（如 `DateTime` vs `TIMESTAMP`），回滚时出现兼容故障。
3. **部署复杂度**: 团队需要同时维护 SQLite 和 PostgreSQL 两套配置。
4. **数据挖掘准确性**: SQLite 的数值精度在 RFM 计算中可能导致舍入误差，影响客户分群结果。

评估选项：

| 选项 | 优势 | 劣势 |
|------|------|------|
| PostgreSQL 16 | 特性完整，FOR UPDATE 支持，丰富的聚合函数，JSON 查询 | 需要独立安装/容器化 |
| MySQL 8 | 社区广泛，pymysql 驱动成熟 | JSON 支持弱于 PG，窗口函数实现较晚 |
| SQLite 保留 | 零配置，本地启动快 | 无并发控制，不支持生产部署 |

## Decision

**统一为 PostgreSQL 16**，并明确以下规范：

1. **docker-compose 生产化**: 使用 `postgres:16-alpine` 镜像，配置健康检查（`pg_isready`），挂载持久化卷 `pgdata`，默认端口 5432。
2. **统一连接串**: 所有环境通过环境变量 `DB_URI` 读取连接串，默认指向 PostgreSQL。示例：`postgresql://shopminer:shopminer@db:5432/shopminer`。
3. **移除 SQLite 配置**: 删除开发环境的 SQLite 回退配置，所有成员在本地通过 Docker 运行 PostgreSQL。
4. **迁移指南就绪**: 编写数据库迁移脚本（`flask db upgrade`），数据初始化通过 `scripts/seed_demo_data.py` 完成。
5. **Alembic 迁移管理**: 所有表结构变更通过 Flask-Migrate 生成迁移文件，提交至版本控制。

## Consequences

**正面**:

- 单一数据库目标消除了方言差异问题。行级锁、日期聚合、JSON 查询等特性在所有环境一致运行。
- docker-compose 配置成为唯一的部署方式，无需为开发环境做额外适配。新成员 `docker compose up -d` 即可启动数据库。
- 利用 PostgreSQL 的 `pg_stat_statements` 可监控慢查询，优化数据挖掘 API 响应时间。
- PostgreSQL 的数值类型（`NUMERIC`、`DOUBLE PRECISION`）保证财务和 RFM 计算的精度。

**负面**:

- 所有开发和 CI 环境必须运行 Docker 或安装 PostgreSQL，无法离线零配置启动。权衡下使用 docker-compose 将数据库容器化，启动成本可接受。
- 数据库迁移从零开始（首次 `flask db upgrade`），无法简单回退到 SQLite 开发模式。
- 本地系统资源占用增加约 256MB（PostgreSQL 容器内存），对低配开发机有影响。

## Compliance

验证方式与示例：

1. **连接串检查**: `app/config.py` 中数据库 URI 以 `postgresql://` 或 `postgresql+psycopg2://` 开头，不存在 `sqlite:///` 或 `mysql://` 配置。
2. **docker-compose 验证**: 根目录 `docker-compose.yml` 中 `db` 服务使用 `image: postgres:16-alpine`，配置健康检查和持久化卷 `pgdata`。
3. **行级锁可运行**: 下单流程（`app/services/order_service.py`）中 `select ... FOR UPDATE` 语句在 PostgreSQL 上正常执行，不抛出 `NotSupportedError`。
4. **迁移文件统一**: Alembic 迁移目录（`migrations/versions/`）下的迁移脚本仅包含 PostgreSQL 兼容的列类型和索引定义。
5. **种子数据脚本**: `scripts/seed_demo_data.py` 通过 `psycopg2` 驱动写入 PostgreSQL，种子数据列与生产表结构一致。
