# ADR-004: 测试策略 — YAML 数据驱动 + Allure 报告 + Playwright POM

## Status

Historical — 项目早期已确立，持续执行中。

## Context

ShopMiner 的测试体系面临规模化挑战。项目包含 443 个测试用例，跨越单元测试（RFM 分群决策表 168 个、边界值 21 个）、API 集成测试（30+ 个场景）和 E2E 测试（5 个核心用户旅程），测试覆盖率达到 82%。

主要问题：

1. **测试数据冗余**: RFM 分群的 168 个决策表用例中，每个用例都内联构造 `(r_score, f_score, m_score) → segment` 的数据。随着分群规则调整，需要逐一修改 168 个测试函数，维护成本极高。
2. **报告可读性差**: `pytest --tb=short` 的输出对开发者友好，但项目管理者或面试演示时缺乏可视化报告。
3. **E2E 维护困难**: Playwright 测试中定位器（`page.locator`）分散在各个测试函数中，页面结构变更时，需要逐一查找和更新。
4. **新用例编写门槛高**: 新成员增加测试需要用 Python 编写完整函数，包括 `assert` 逻辑和异常处理，对非 Python 背景的团队成员不友好。

评估方案：

| 维度 | 原方案 | 新方案 |
|------|--------|--------|
| 测试数据管理 | Python 内联字典 | YAML 外部数据文件 |
| 报告 | pytest CLI 输出 | Allure Framework |
| E2E 模式 | 函数式 Playwright | Page Object Model |
| 前端测试 | 无 | Vitest + @vue/test-utils |

## Decision

采用 **YAML 数据驱动测试 + Allure 报告 + Playwright POM** 的综合测试策略：

### 1. YAML 数据驱动测试

单元和集成测试的输入数据和预期结果存储在 YAML 文件中，测试框架通过参数化注入：

```
tests/
  data/
    rfm_segments.yaml        # RFM 分群决策表 168 场景
    boundary_values.yaml     # 价格/库存/整数边界 21 场景
    bug_regression.yaml      # 7 个历史 Bug 回归数据
```

YAML 数据格式示例：

```yaml
# tests/data/rfm_segments.yaml
test_cases:
  - name: "高价值客户_R5_F5_M5"
    r: 5
    f: 5
    m: 5
    expected: "高价值客户"
  - name: "流失预警_R2_F5_M1"
    r: 2
    f: 5
    m: 1
    expected: "流失预警"
```

测试代码通过 `pytest.mark.parametrize` 读取 YAML：

```python
@pytest.mark.parametrize("case", load_yaml("rfm_segments.yaml"))
def test_rfm_segment(case):
    result = _compute_segment(case["r"], case["f"], case["m"])
    assert result == case["expected"]
```

### 2. Allure 报告

- 使用 `--alluredir=tests/report/allure-results` 收集测试结果。
- `pytest.ini` 中已配置 `--alluredir=tests/report/allure-results` 为默认参数。
- 整合 Allure 的步骤注解（`@allure.step`）、严重级别（`@allure.severity`）和 epic/feature/story 标签。
- 生成 HTML 报告供面试展示和项目评审使用。

### 3. Playwright Page Object Model

E2E 测试从函数式迁移到 POM 模式：

```
tests/e2e/
  pages/
    login_page.py        # 登录页 POM
    product_page.py      # 商品列表页 POM
    cart_page.py         # 购物车页 POM
    checkout_page.py     # 下单页 POM
    admin_page.py        # 管理后台 POM
  test_e2e_flow.py       # 核心用户旅程测试
```

每个 Page Object 封装页面的定位器和操作：

```python
class LoginPage:
    def __init__(self, page):
        self.page = page
        self.email_input = page.locator('[data-testid="email-input"]')
        self.password_input = page.locator('[data-testid="password-input"]')
        self.submit_btn = page.locator('[data-testid="login-submit"]')

    def login(self, email, password):
        self.email_input.fill(email)
        self.password_input.fill(password)
        self.submit_btn.click()
```

### 4. 前端单元测试

使用 Vitest + @vue/test-utils 测试 Vue 组件逻辑：

```
frontend/
  src/components/__tests__/
    ProductCard.spec.js
    CartItem.spec.js
  src/stores/__tests__/
    cartStore.spec.js
```

## Consequences

**正面**:

- **新增测试 90% 只改 YAML**: 分群规则变化时，只需编辑 `rfm_segments.yaml` 中的映射关系，无需改动 Python 代码。边界值用例同样只编辑数据文件。
- **报告清晰**: Allure HTML 报告提供时序图、分类汇总、失败截图（Playwright）等丰富信息，适合面试展示和 stakeholder review。
- **定位器集中管理**: POM 模式将 Playwright 定位器集中在 Page 类中，页面结构变更时只需修改一处。
- **前端测试覆盖**: Vitest 在构建流水线中自动运行，确保 Vue 组件和 Pinia store 变更不引入回归。

**负面**:

- **YAML 框架需要维护**: 需要维护 `load_yaml()` 工具函数、数据文件组织和校验逻辑。YAML 文件的拼写错误只有在测试运行时才能发现。
- **Allure 额外步骤**: 需要启动 Allure 服务生成报告（`allure serve tests/report/allure-results`），CI 中需额外安装 `allure-commandline`。
- **POM 初期工作量**: 初次编写 Page Object 类比直接写 Playwright 代码耗时长。但 E2E 场景数量不多（5 个），总体成本可控。
- **双测试框架**: 前端（Vitest）和后端（pytest）使用不同测试运行器，CI 配置中需分别调用。

## Compliance

验证方式与示例：

1. **YAML 数据文件**: `tests/data/` 目录下存在 `rfm_segments.yaml`、`boundary_values.yaml`、`bug_regression.yaml` 至少三个数据文件。
2. **数据驱动测试**: 使用 `ast_grep_search` 搜索 `load_yaml(` 或 `@pytest.mark.parametrize` 配合 YAML 加载的测试函数。RFM 分群测试应不从内联数据读取，全部从 YAML 注入。
3. **Allure 配置**: `pytest.ini` 中包含 `--alluredir=tests/report/allure-results`。`pyproject.toml` 或 `requirements.txt` 中包含 `allure-pytest` 依赖。
4. **Playwright POM**: `tests/e2e/pages/` 目录下至少包含 `login_page.py`、`product_page.py`、`cart_page.py`、`checkout_page.py`、`admin_page.py` 五个 Page Object 文件。
5. **前端测试**: `frontend/package.json` 中 `scripts` 包含 `"test:unit": "vitest run"`。`frontend/src/` 下存在 `__tests__` 目录。
6. **覆盖率**: 运行 `pytest --cov=app` 时总体覆盖率不低于 80%，RFM 模块覆盖率不低于 90%。
