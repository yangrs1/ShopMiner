# ADR-003: 将 ML 管道从 subprocess 迁移到 Celery+Redis

## Status

Proposed — 设计已确定，待实施。

## Context

ShopMiner 包含四个数据挖掘模块：RFM 客户分群、销量预测、流失预警、关联规则挖掘。这些计算由 `scripts/compute_analytics.py` 统一管理，当前实现通过 `subprocess.Popen` 以阻塞方式执行：

```python
# app/services/analytics_service.py:trigger_recompute()
proc = subprocess.Popen(
    [sys.executable, script_path, "--force"],
    stdout=subprocess.PIPE, stderr=subprocess.PIPE,
    cwd=project_root,
)
```

该方案暴露以下问题：

1. **管理后台卡顿**: `trigger_recompute()` 在 API 请求线程中启动子进程，虽使用 `Popen`（非阻塞调用），但后续查询状态、读取结果等操作仍占用请求-响应循环。当数据量大时（如全量 RFM 计算涉及数万用户），API 响应时间可超过 30 秒。
2. **任务状态不可追踪**: `subprocess.Popen` 返回后，无法查询进度、取消任务、重试失败或记录执行历史。用户无法知道"重算进行到哪一步"。
3. **并发安全性差**: 多次触发重算会启动多个并行子进程，写入同一张表时产生竞态条件。MFG 计算可能读到不完整的数据。
4. **错误处理弱**: 子进程的标准错误输出仅在失败时被记录，但没有重试机制和告警。

评估方案：

| 方案 | 复杂度 | 任务追踪 | 可观测性 |
|------|--------|----------|----------|
| Celery + Redis | 中 | 完善（花、状态、结果后端） | Flower UI |
| APScheduler | 低 | 无任务状态追踪 | 日志 |
| 独立 ML 微服务 | 高 | 需自建队列和状态管理 | 自定义 |
| 保持 subprocess | 无 | 无 | 无 |

## Decision

采用 **Celery + Redis** 异步任务架构，具体方案如下：

1. **新增 Celery Worker**: 项目新增 `app/tasks/celery_app.py`，配置 Celery 应用连接到 Redis 作为消息代理（Broker），PostgreSQL 作为结果后端（Result Backend）。
2. **新增 Redis 服务**: `docker-compose.yml` 增加 `redis` 服务，使用 `redis:7-alpine` 镜像。
3. **任务拆分**: 将 `compute_analytics.py` 中的四个计算模块拆分为独立 Celery 任务：
   - `tasks.compute_rfm()` — RFM 客户分群
   - `tasks.compute_sales_forecast()` — 销量预测
   - `tasks.compute_churn_prediction()` — 流失预警
   - `tasks.compute_association_rules()` — 关联规则
4. **编排任务**: 新增 `tasks.orchestrate_recompute()` 编排任务，按依赖顺序串行执行（RFM → 销量 → 流失 → 关联），任一任务失败则停止并记录。
5. **API 改造**: `trigger_recompute()` 改为提交编排任务并返回 `task_id`，新增 `GET /api/v1/analytics/task/{task_id}` 查询任务状态。

新增基础设施：

```
docker-compose.yml 新增服务:
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
```

新增环境变量：

```
CELERY_BROKER_URL=redis://redis:6379/0
CELERY_RESULT_BACKEND=db+postgresql://shopminer:shopminer@db:5432/shopminer
```

## Consequences

**正面**:

- **异步非阻塞**: ML 计算完全脱离 HTTP 请求-响应循环，管理后台不再卡顿。用户触发重算后立即获得 `task_id`，后续轮询状态。
- **任务状态可追踪**: Celery 提供 `PENDING → RECEIVED → STARTED → SUCCESS/FAILURE` 完整状态机，配合 `task_id` 可精确查询进度、结果和错误信息。
- **可观测性提升**: 集成 Flower（`flower:5555`）实时监控 Worker 池、队列深度、任务耗时和失败率。
- **并发安全**: 编排任务保证同一时间只有一个重算流程运行，通过 `task.once` 或 Redis 锁防止重复触发。
- **可扩展**: 未来可增加 Worker 节点水平扩展，或按任务优先级分配不同队列。

**负面**:

- **基础设施复杂度增加**: 新增 Redis 服务需要维护，docker-compose 从 2 个服务变为 3 个（web + db + redis），CI 流水线需额外配置。
- **内存开销**: Redis 容器约占用 50-100MB 内存，低配服务器上需要评估。
- **状态一致性**: Celery task 执行过程中服务宕机，可能导致"已跑完 RFM 但还没跑销量"的不一致状态。需实现幂等性和定期清理中间状态。
- **学习成本**: 团队成员需要熟悉 Celery 的概念（Worker、Broker、Beat、Result Backend）。

## Compliance

验证方式与示例：

1. **Redis 运行**: `docker-compose.yml` 中存在 `redis` 服务配置，使用 `redis:7-alpine` 镜像，暴露 6379 端口。运行 `docker compose ps` 可验证 redis 服务状态为 healthy。
2. **Celery 应用初始化**: `app/tasks/celery_app.py` 中存在 `celery = Celery('shopminer', broker=os.getenv('CELERY_BROKER_URL'))` 声明。
3. **任务注册**: 四个计算任务（`compute_rfm`、`compute_sales_forecast`、`compute_churn_prediction`、`compute_association_rules`）均使用 `@celery.task` 装饰器注册。
4. **API 返回 task_id**: 调用 `POST /api/v1/analytics/recompute` 返回 `{ "status": "started", "task_id": "..." }`，不再返回 `pid`。
5. **无 subprocess 调用**: `analytics_service.py` 中不再导入 `subprocess` 模块，`trigger_recompute()` 函数改为 `from app.tasks import orchestrate_recompute` 提交任务。
6. **任务状态查询**: `GET /api/v1/analytics/task/{task_id}` 返回 `AsyncResult.state` 和 `AsyncResult.info`。
7. **集成测试**: 测试中通过 `celery_app.conf.task_always_eager = True` 同步执行任务，验证计算逻辑正确性。
