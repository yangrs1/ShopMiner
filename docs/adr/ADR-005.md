# ADR-005: 前端技术栈 — Vue 3 + Element Plus + ECharts

## Status

Historical — 项目前期已确定，未发生变更。

## Context

ShopMiner 的前端需要同时满足三类页面场景：

1. **电商核心流程**: 商品浏览、购物车、下单、支付、订单管理。需要组件成熟、交互流畅的 UI 框架。
2. **数据挖掘可视化**: RFM 散点图、销量趋势折线图、关联规则网络图、流失分析柱状图。需要功能强大的图表库。
3. **管理后台**: 商品管理、订单管理、数据大屏、定价建议。需要开箱即用的表格、表单、对话框组件。

此外，项目使用 Flask 作为后端，前端采用前后端分离模式，通过 REST API 通信。团队在 Vue 2 和 Element UI 上有既有经验。

评估方案：

| 方案 | UI 组件 | 可视化 | 状态管理 | 测试 | 优势 | 劣势 |
|------|---------|--------|----------|------|------|------|
| Vue 3 + Element Plus | Element Plus | ECharts | Pinia | Vitest | 组件丰富，过渡平滑 | 无 TypeScript 原生支持 |
| React + Ant Design | Ant Design | ECharts | Redux | Jest | 生态最大 | 团队 React 经验不足 |
| React + Material UI | MUI | Recharts | Zustand | Vitest | 设计系统完善 | MUI API 学习成本高 |
| Svelte + Skeleton | Skeleton | Chart.js | Svelte store | Vitest | 编译后体积小 | 社区资源有限 |

## Decision

选用 **Vue 3 (Composition API) + Element Plus + ECharts + Pinia + Vitest** 作为前端技术栈：

### 框架选型

- **Vue 3 + Composition API**: 利用 `setup()` 语法糖组织组件逻辑，`ref`/`reactive` 管理响应式状态，`computed` 处理派生数据，`watch` 监听数据变化。
- **Element Plus**: Element UI 的 Vue 3 版本，提供 `el-table`、`el-form`、`el-dialog`、`el-menu` 等常用组件。组件库统一了管理后台和商家页面的视觉风格。
- **ECharts 5**: 通过 `echarts` npm 包直接集成，RFM 散点图使用 `scatter` 系列，销量趋势使用 `line` 系列，关联规则使用 `graph` 系列，流失分析使用 `bar` 系列。
- **Pinia**: 替代 Vuex，用于管理跨组件共享状态（购物车、用户信息、主题配置）。
- **Vue Router 4**: 前端路由，支持懒加载和导航守卫（认证检查）。
- **Axios**: HTTP 请求客户端，封装统一请求拦截器（自动附加 JWT token）和响应拦截器（统一错误处理）。

### 构建与测试

- **Vite 5**: 构建工具，开发服务器支持 HMR 热更新。
- **Vitest**: 单元测试框架，兼容 Jest 语法，与 Vite 共享配置。
- **@vue/test-utils 2**: Vue 3 官方测试工具库。

### 关键决策说明

- **为什么没有使用 TypeScript？** 团队保持纯 JS，出于两个原因：(1) 项目核心同学的前端背景以 JS 为主，引入 TS 会增加学习和迁移时间；(2) 项目规模为中型 SPA，类型系统带来的收益与投入不成正比。Vue 3 的 Composition API 配合 JSDoc 注释已提供足够的内省能力。
- **为什么没有使用 Nuxt？** 项目是 Flask 提供后端 + Vue SPA（静态文件由 Flask 分发），不需要服务端渲染。Nuxt 的 SSR、SSG 等能力在本项目中用不上。

## Consequences

**正面**:

- **组件化开发**: Element Plus 的 60+ 组件覆盖了电商和管理后台的绝大多数场景，无需自建 UI 组件。
- **图表丰富**: ECharts 的 20+ 图表类型和丰富的交互能力（tooltip、dataZoom、legend）使数据挖掘结果的展示直观且可探索。
- **状态管理简洁**: Pinia 相比 Vuex 减少了样板代码，`defineStore` 语法与 Composition API 风格一致。
- **快速开发迭代**: Vite HMR 使得前端修改后秒级刷新，与 Flask API 联调效率高。`vite.config.js` 配置代理指向 Flask 开发服务器。
- **测试集成方便**: Vitest 与 Vite 共享 `vite.config.js` 配置，无需 Jest 的额外配置。

**负面**:

- **无 TypeScript**: 大型组件（如管理后台的商品管理页）中，props 和复杂对象缺乏类型约束，重构时容易遗漏字段。项目中通过 JSDoc 部分缓解，但不是完全替代方案。
- **包体积较大**: Element Plus 按需引入后仍有约 300KB（gzip 后约 80KB），ECharts 全量引入约 1MB（gzip 后约 300KB）。对于 SPA 应用可接受，但首屏加载时间受一定影响。
- **Vue 3 迁移成本**: 部分第三方库（如 vue-clipboard2）尚未完全适配 Vue 3，改用原生 API 或 fork 版本。
- **无 SSR**: 不利于 SEO，但 ShopMiner 是后台型应用而非内容型网站，SEO 不是关键需求。

## Compliance

验证方式与示例：

1. **package.json 依赖检查**: `frontend/package.json` 的 `dependencies` 中包含 `vue`（^3.4.0）、`element-plus`（^2.7.0）、`echarts`（^5.5.0）、`pinia`（^2.1.0）、`vue-router`（^4.3.0）、`axios`（^1.7.0）。
2. **构建脚本**: `frontend/package.json` 的 `scripts` 中包含 `"build": "vite build"`。运行 `npm run build` 后，`frontend/dist/` 目录生成 `index.html` 和 `static/` 子目录。
3. **TypeScript 排除**: `frontend/` 下不存在 `tsconfig.json` 文件，Vue 组件使用 `.js` 后缀而非 `.ts`。
4. **Pinia Store 存在**: `frontend/src/stores/` 目录下至少包含 `user.js`、`cart.js` 等 Pinia store 定义文件，使用 `defineStore` 语法。
5. **Vue Router 配置**: `frontend/src/router/index.js` 中存在路由表定义，包含 `meta: { requiresAuth: true }` 等守卫配置。
6. **ECharts 实例化**: 管理后台的 RFM 页面、销量页面、关联规则页面中通过 `echarts.init()` 创建图表实例，并在 `onUnmounted()` 中调用 `dispose()` 清理。
7. **Vitest 可运行**: `cd frontend && npm run test:unit` 可正常执行，输出测试通过/失败结果。
