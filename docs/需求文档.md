# ShopMiner 模型优化需求文档

## 1. 背景
ShopMiner 电商数据挖掘平台已完成 Phase1-5 基础建模，但当前模型指标存在以下问题：
- **流失预警**：CV F1=0.8063 vs Validation OOT F1=0.6245，差距 0.18，严重过拟合
- **聚类**：Silhouette≈0.174 (WEAK)，分群边界模糊
- **销售预测**：MAPE≈15.17%，预测偏差较大
- **概率校准**：CalibratedClassifierCV 与 XGBoost float32 不兼容，概率不可靠

## 2. 核心目标
在不改变前端接口和数据表结构的前提下，提升模型质量指标，使前端展示数据更可信。

## 3. 技术栈
- Python 3.12
- XGBoost, LightGBM, Prophet
- scikit-learn, imbalanced-learn, optuna
- pandas, numpy, matplotlib

## 4. 交付物
- `phase4_churn_v3.py` — 流失预警优化脚本
- `phase3_clustering_v2.py` — 聚类优化脚本
- `phase5_prophet_v2.py` — 销售预测优化脚本
- `sensitivity_analysis_v2.py` — 敏感性分析优化脚本
- 各模型评估图表 (PNG)

## 5. 绝对不做清单
- 不修改前端 Vue 代码
- 不修改后端 API 接口
- 不修改数据库表结构
- 不修改 compute_analytics.py 的输入输出格式
- 不添加未明确要求的功能

### 6.3 关键改进点

#### Phase 3 聚类
- **方法**：log1p 变换 + MinMaxScaler + MiniBatchKMeans K=4
- **原理**：右偏 RFM 数据 log 变换后距离度量合理；MinMax 比 Standard 适合 K-Means
- **Optuna 二次调参**：Silhouette 0.360 → 0.360（边际 +0.08%，验证已接近 KMeans 极限）

#### Phase 4 流失
- **方法**：XGBoost + scale_pos_weight（v6），Optuna 精调 9 个超参（v7）
- **关键发现**：v5 Stacking Test AUC 0.918 但 OOT 0.631（严重过拟合）；Optuna 改用**时序 holdout** objective 后找到 v7 = OOT 0.913
- **SHAP 验证**：v7 的 53 特征每个都有贡献，K=20 选特征会掉 1.5% OOT

#### Phase 5 销售
- **方法**：LightGBM weekly + 11 个 UK 日历特征
- **Optuna 失败**：TimeSeriesSplit CV-5 找的浅树（max_depth=3）在 8 周 test 上退化 10%
- **Darts N-BEATS 失败**：80 周训练样本太小，DL SMAPE 15.5% vs LGBM 4.86%
- **结论**：v3 已是此数据集的天花板

#### Phase 6 关联
- **方法**：FP-Growth min_sup=0.008 max_len=4 + 分群规则
- **Optuna 退化**：找到的 392 规则 @ lift 17 业务覆盖不如 v3 的 694 规则 @ lift 11

## 7. v3 进展更新（2026-06-01）— 模型可视化

### 7.1 需求背景
Admin 管理后台的"模型指标"页面原只展示文本指标表，业务视角不直观。
用户需要：1 个核心业务图（主页面）+ 2-3 个全量技术图（详情页弹窗）

### 7.2 交付清单
- `scripts/generate_model_viz.py` — 4 模型图表数据预计算（venv 跑）
- `app/services/analytics_service.py:530` 新增 `get_model_viz()` — 读取 JSON 服务
- `app/api/v1/analytics/routes.py:381` 新增 `GET /api/v1/analytics/viz/<model>` — 4 端点
- `frontend/src/api/index.js:129` 新增 `getModelViz()` — 前端 API
- `frontend/src/views/Admin.vue` — 4 个主页面核心图 + 4 个 el-dialog 弹窗全量图
- `scripts/compute_analytics.py:80-95` — 重算流程末尾自动调 viz 生成

### 7.3 预计算数据清单

| 模型 | 主页面核心图（业务） | 详情页弹窗全量图（技术） |
|---|---|---|
| Phase 3 聚类 | 簇规模分布柱状图 | PCA 2D 散点 + 簇 RFM 画像 + 簇业务表 |
| Phase 4 流失 | OOT 滚动窗口 AUC 折线 | ROC 曲线 + OOT 时序柱状 + Top 15 特征重要性 |
| Phase 5 销售 | 实际 vs 预测 8 周对比 | 实际vs预测+误差 + 残差直方 + 月度季节性 |
| Phase 6 关联 | Top 10 高 Lift 规则 | Top 20 规则 + 支持度vs置信度散点 + 分群规则统计 |

### 7.4 设计原则
- **预计算原则**：数据写入 `data/prep/phase*_viz.json`，用户看页面 0 等待
- **管理员触发**：只点"重新计算"才重算，4 模型共 <15s
- **设计双视角**：主页面业务视角（KPI 卡 + 1 图），详情页技术视角（元数据 tag + 2-3 图）
- **不改前端结构**：Admin.vue 复用现有 model-header，添加 viz section + el-dialog
